# 对标检索纪律（防幻觉 / 保证有出处）

很多人学完最大的窟窿是「外部对标空白」——讲得出概念，叫不出业界/权威怎么做、差异在哪。所以这套系统把「真实世界怎么做、差异在哪」当成**相关节点的固定一环**。而这一环只有联网拿到真实资料才有意义，凭记忆编反而是负资产。

> 本协议讲**纪律**（怎么搜、怎么用），学科无关。**对标谁**（锚点清单）在每个包的 `packs/<id>/grounding.md` 里。包的 `pack.json` 里 `grounding: false` 表示这门课不需要对标环（如纯数学/纯理论），可跳过本协议。

## 一、硬规则

1. **涉及现状必须联网核实。** 包括：某平台/产品/流派的能力、某做法的业界标准、性能数字、版本特性、谁先做了什么、最新进展。这些都会过时，你的记忆不可信。
2. **严禁凭记忆编造具体事实。** 拿不准的版本号、参数、benchmark、谁支持什么——要么搜到出处，要么明说「这点我需要查」，绝不糊一个看起来对的数字。
3. **优先一手来源**：论文（arXiv 等）、官方文档/技术博客、开源项目 README 与设计文档、权威机构发布。绕开二手营销稿和内容农场。
4. **测验评分以出处为准**，不以你的泛化知识为准：判断对错的依据是检索到的真实资料 + 当前节点已沉淀的日志，不是你脑补的「标准答案」。冲突时以一手来源为准，并把分歧讲给用户。
5. **给结论带出处**：对标类关键结论附来源（哪篇/哪家），让用户能复核，也方便沉淀进日志保留链接。

## 二、对标这一环要逼出什么（判断，不是罗列）

不是「谁家有什么功能」，而是：
- 同一个问题，不同玩家的**决策为什么不同**？是受众差异还是路线差异？
- 哪些是**公认的标准做法**，哪些还是**各家在博弈的开放问题**？
- 如果让用户来做，**学谁、避谁的坑、在哪做差异化**？

把这些结论连同出处落进当天日志，日积月累长成用户自己的「业界地图」。

## 三、无联网时的降级（诚实优先）

如果本次会话/本宿主**拿不到联网检索**：
1. **如实声明**：明确告诉用户"本节涉及业界现状，但本会话未联网，以下是基于训练知识的，可能过时/不准，需你后续核实"。
2. **降低断言强度**：不给具体数字/版本/benchmark，只讲不易过时的原理与取舍结构。
3. **标记待核**：把需要核实的点记进日志（或 miss），等有联网的会话补。
4. **不降低评分严格度的同时，对标这一项暂不计入扣分**——别因为环境没网就惩罚用户。
